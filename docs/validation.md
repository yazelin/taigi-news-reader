## `0.1.2` private-test 實作狀態

`0.1.2` source 已實作逐人 invite-token authentication／quota。Raw token 只存 Chrome local settings、綁 configured origin、只送同 origin `/v1/`；server config 只有 SHA-256 digest＋stable pseudonymous subject；SQLite 只持久保存當日 `utc_date/subject/jobs/characters`；jobs 依 subject ownership，terminal result 使用 one-shot delivery lease，並有 daily／outstanding／bytes caps。Private beta 因 job registry 是 process-local，只能使用單一 backend worker。

Repo automated baseline 已更新為 backend `166 passed`、extension `82/82`＋ESLint／production build，並產生 `taigi-news-reader-0.1.2.zip`（50,789 bytes，SHA-256 `5639d9b33090a50470dd800ce03c2c620d55fbadea3b4f821c1ab119b6e012e6`）。尚未完成的是 fresh-profile exact-package Chrome E2E、private-beta ingress 在 `.11` 的 live deployment、外部 reviewer reachability／abuse smoke、Groq ZDR、曾曝光 provider key rotation與CWS `0.1.2` upload；完成並留下不含 secrets／個資的證據前不可送審。

上述 Groq smoke 證明的是「OpenAI-compatible 翻譯 → gate-valid POJ → MMS TTS → 可解碼 WAV」在這兩次實際請求中完整跑通。POJ 字元 gate 與 WAV 格式檢查只能驗證輸入字表及音訊容器，**不代表翻譯內容、台語用詞、發音或長輩可懂度已通過**。LLM provider 的輸出具有非決定性；即使輸入、模型名稱與參數相同，後續請求仍可能產生不同結果，因此不能把單次 smoke 當成永久品質保證。

有介面重測也實際觀察到這項非決定性：80 字新聞曾因 comma 與 superscript n 這類不發音排版字元在 strict repair 後仍無法通過 MMS 字元 gate。後端當時會明確回錯而不是合成不可信輸出；現已用 deterministic formatting normalization 消除這個特定 failure mode，並以同文後續 production job 通過驗證。GPT-OSS 仍使用 low reasoning 並保留 8,192 completion tokens；首次 translation content 為空時也會用完全相同的 provider request 自動重試一次，不會拿 reasoning、華語原文或其他 provider 冒充翻譯。這些修正改善 formatting／transport reliability，但不保證每次翻譯的語意或台語品質都正確。
